06 · Authentication Flow

Twouple uses a phone-based OTP login flow. No passwords needed. The flow provisions new users automatically on first login (findOrCreate pattern).

1
User enters phone number

Client sends POST /api/v1/auth/request-otp with the phone number.

2
Server dispatches OTP

In production, an SMS is sent via Twilio/AWS SNS. In dev mode, the OTP is hardcoded as 1234.

3
User submits OTP

Client sends POST /api/v1/auth/verify-otp with phone + OTP code.

4
Server verifies & provisions

OTP is validated → User is found or created → JWT token signed with 30-day expiry.

5
Client stores JWT

The returned JWT is stored locally. All subsequent requests include it via Authorization: Bearer <token>.

✅ Dev Tip: In development mode (USE_REAL_OTP=false), use OTP code 1234 for all phone numbers.

07 · Auth Middleware

The protect middleware guards all private endpoints. It extracts and verifies the JWT from the Authorization header, confirms the user still exists in the DB, and attaches req.user for downstream handlers.

How to use

All routes in /users, /relationships, /chat, and /insights are wrapped with router.use(protect) — every request must include a valid JWT.

Request Header Format

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Error Responses

ScenarioStatusMessage
No token provided401Access denied. No token provided.
Token expired401Session expired. Please log in again.
Invalid/tampered token401Invalid token.
User deleted after token issued401The user belonging to this token no longer exists.

09 · Auth API Endpoints

Phone-based OTP authentication. No passwords, no emails — just a phone number and a verification code.

POST /api/v1/auth/request-otp 🌐 PUBLIC

Sends an OTP code to the provided phone number. In development mode, the OTP is always 1234. In production, this would integrate with Twilio or AWS SNS.

Request Body

FieldTypeRequiredDescription
phone_number string Required User's phone number (min 10 characters)

Example Request

POST /api/v1/auth/request-otp
{
  "phone_number": "+919876543210"
}

Success Response — 200 OK

{
  "success": true,
  "message": "OTP dispatched successfully"
}

Error Responses

400 — Invalid phone number 500 — Internal server error
POST /api/v1/auth/verify-otp 🌐 PUBLIC

Verifies the OTP, creates the user if they don't exist (find-or-create pattern), and returns a signed JWT token with a 30-day expiry. Also returns whether this is a new user (for onboarding flow).

Request Body

FieldTypeRequiredDescription
phone_number string Required The phone number that received the OTP
otp string Required The OTP code (use 1234 in dev)

Example Request

POST /api/v1/auth/verify-otp
{
  "phone_number": "+919876543210",
  "otp": "1234"
}

Success Response — 200 OK

{
  "success": true,
  "message": "Authentication successful",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "isNewUser": true,
    "user": {
      "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "phone_number": "+919876543210",
      "is_premium": false
    }
  }
}

Error Responses

400 — Missing phone or OTP 401 — Invalid OTP 500 — Internal server error

10 · User / Profile API Endpoints

All user endpoints require authentication. The user ID is extracted from the JWT token automatically — users can only access their own data.

🔒 Auth Required: All endpoints below require Authorization: Bearer <token>
GET /api/v1/users/me 🔒 AUTH

Fetch the authenticated user's profile. Excludes internal timestamps from response.

Success Response — 200 OK

{
  "success": true,
  "data": {
    "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "phone_number": "+919876543210",
    "full_name": "Arjun Mehta",
    "dob": "1998-03-15",
    "time_of_birth": "14:30",
    "place_of_birth": "Mumbai, Maharashtra",
    "zodiac_sign": "Pisces",
    "is_premium": false
  }
}

Error Responses

401 — Unauthorized 404 — User not found
PUT /api/v1/users/me 🔒 AUTH

Update the authenticated user's profile. Primarily used for the onboarding flow after first login. All fields are optional — send only what you want to update. Validated with Joi.

Request Body

FieldTypeRequiredValidation
full_name string Optional 2–100 characters
dob date Optional ISO format: YYYY-MM-DD
time_of_birth string Optional Format: HH:MM (24hr)
place_of_birth string Optional Max 255 characters
zodiac_sign enum Optional One of: Aries, Taurus, Gemini, Cancer, Leo, Virgo, Libra, Scorpio, Sagittarius, Capricorn, Aquarius, Pisces

Example Request

PUT /api/v1/users/me
{
  "full_name": "Arjun Mehta",
  "dob": "1998-03-15",
  "time_of_birth": "14:30",
  "place_of_birth": "Mumbai, Maharashtra",
  "zodiac_sign": "Pisces"
}

Success Response — 200 OK

{
  "success": true,
  "message": "Profile updated successfully",
  "data": { /* full updated user object */ }
}

Error Responses

400 — Validation error 401 — Unauthorized 500 — Failed to update
DELETE /api/v1/users/me 🔒 AUTH

Permanently deletes the authenticated user's account. This is a hard delete required for Apple App Store and Google Play policy compliance. This action is irreversible.

Success Response — 200 OK

{
  "success": true,
  "message": "Account permanently deleted"
}
⚠️ Irreversible: This permanently deletes the user record. There is no soft-delete or recovery mechanism.

11 · Relationship API Endpoints

Manage partner invitations, acceptance, and relationship status. A user can only be in one active or pending relationship at a time.

🔒 Auth Required: All endpoints below require authentication.
POST /api/v1/relationships/invite 🔒 AUTH

Invite a partner by their phone number. Creates a pending relationship. The partner must already be registered on Twouple. Self-invites and duplicate relationships are blocked with validation.

Request Body

FieldTypeRequiredDescription
partner_phone string Required Partner's registered phone (10–15 chars)

Example Request

POST /api/v1/relationships/invite
{
  "partner_phone": "+919123456789"
}

Success Response — 200 OK

{
  "success": true,
  "message": "Invitation sent successfully.",
  "data": {
    "id": "rel-uuid-here",
    "user_one_id": "inviter-uuid",
    "user_two_id": "partner-uuid",
    "status": "pending",
    "composite_score": null,
    "established_at": null
  }
}

Error Responses

400 — Self-invite / already in relationship 400 — Partner not registered
GET /api/v1/relationships/status 🔒 AUTH

Get the current relationship status for the authenticated user. Returns partner details if a relationship exists. Status can be none, pending, active, or disconnected.

Success Response — 200 OK

{
  "success": true,
  "data": {
    "status": "active",
    "data": {
      "id": "rel-uuid",
      "status": "active",
      "PartnerOne": { "id": "...", "full_name": "Arjun", "phone_number": "..." },
      "PartnerTwo": { "id": "...", "full_name": "Priya", "phone_number": "..." }
    }
  }
}
POST /api/v1/relationships/accept 🔒 AUTH

Accept a pending invitation. Only the invited user (user_two) can call this. Changes status from pending → active and records established_at timestamp.

Success Response — 200 OK

{
  "success": true,
  "message": "Relationship established!",
  "data": {
    "id": "rel-uuid",
    "status": "active",
    "established_at": "2026-04-19T17:30:00.000Z"
  }
}

Error Responses

400 — No pending invitation found

12 · Chat API Endpoints

In-app messaging between partners. Messages are tied to the active relationship and support pagination. The system also supports AI-generated messages (via the is_ai flag).

POST /api/v1/chat/send 🔒 AUTH

Send a message within the user's active relationship. The user must be in an active relationship to send messages.

Request Body

FieldTypeRequiredValidation
message_body string Required 1–2000 characters

Example Request

POST /api/v1/chat/send
{
  "message_body": "Hey love, how's your day going? ❤️"
}

Success Response — 201 Created

{
  "success": true,
  "data": {
    "id": "msg-uuid",
    "relationship_id": "rel-uuid",
    "sender_id": "user-uuid",
    "message_body": "Hey love, how's your day going? ❤️",
    "is_ai": false,
    "createdAt": "2026-04-19T17:30:00.000Z"
  }
}

Error Responses

400 — Validation failed 403 — Not in active relationship
GET /api/v1/chat/history 🔒 AUTH

Retrieve paginated chat history for the active relationship. Messages are ordered newest-first. Includes sender info (id and full_name) for each message.

Query Parameters

ParamTypeDefaultDescription
page integer 1 Page number for pagination
limit integer 50 Messages per page

Example Request

GET /api/v1/chat/history?page=1&limit=20
No request body required

Success Response — 200 OK

{
  "success": true,
  "data": {
    "total_messages": 128,
    "total_pages": 7,
    "current_page": 1,
    "messages": [
      {
        "id": "msg-uuid",
        "message_body": "Thinking of you 💜",

17 · Middlewares Reference

authMiddleware.js — protect

Extracts the JWT token from the Authorization: Bearer header, verifies it cryptographically, confirms the user exists in the DB, and attaches req.user (with id, phone_number, is_premium) for downstream handlers.

errorMiddleware.js — notFound + errorHandler

notFound: Catches any request that doesn't match a defined route and returns a 404 with the attempted URL.
errorHandler: Global error catcher. Returns error message and stack trace (dev only). Falls back to HTTP 500.

validation.js — validateRequest(schema)

A higher-order function that takes a Joi schema and returns Express middleware. Features:

  • abortEarly: false — returns ALL validation errors, not just the first
  • stripUnknown: true — silently removes unexpected fields from the request body
  • Replaces req.body with sanitized values after validation

rateLimiter.js

Rate limiting middleware using express-rate-limit. See the Rate Limiting section below for details.

18 · Utility Modules

locationFormatter.js — getCoordinates(cityName)

Geocoding utility that converts a city name to latitude/longitude using the Nominatim (OpenStreetMap) API. Returns { lat, lng, formatted_name }. Used for astrological birth chart calculations. Production would swap this for Google Maps Geocoding API.

logger.js

Environment-aware logging utility with three levels:

  • logger.info(message, meta) — 🔵 Blue info logs
  • logger.warn(message, meta) — 🟠 Orange warning logs
  • logger.error(message, errorObj) — 🔴 Red error logs with stack trace

In development: colored emoji-prefixed console output.
In production: structured JSON output for log aggregation tools (CloudWatch, Datadog, etc.).

whatsappParser.js — cleanWhatsAppChat(rawText)

Cleans and normalizes exported WhatsApp chat text for AI analysis. Features:

  • Strips WhatsApp timestamp patterns (multiple formats)
  • Replaces media omitted tags with readable labels
  • Removes empty lines and trims whitespace
  • Token safeguard: caps output at ~3000 words (most recent) to fit within AI context windows

19 · Global Error Handling

The API uses a two-layer error catching approach attached at the end of the Express middleware chain.

Standard Error Response Format

{
  "success": false,
  "message": "Error description here",
  "stack": "Error stack trace (dev only)"
}

Error Flow

1
Route not matched

notFound middleware catches it → sets status 404 → passes to errorHandler

2
Error thrown in handler

errorHandler catches it → logs error → returns JSON with status code and message

3
Stack trace visibility

Stack traces are only included in the response when NODE_ENV === 'development'

20 · Rate Limiting Strategy

Two tiers of rate limiting protect the API from abuse and control costs on expensive operations.

🟢
Standard Limiter

100 requests / 15 minutes

For general endpoints: profile, status, chat history

🔴
Strict Limiter

5 requests / 1 hour

For expensive operations: OTP requests, AI endpoints

Rate Limit Error Response

{
  "success": false,
  "message": "Too many requests from this IP, please try again after 15 minutes."
}

Rate limit headers are included in responses via standardHeaders: true (RFC 6585 compliant).
